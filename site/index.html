The Standard

Curated Data Standard (CDS)

A W3C Verifiable Credentials 2.0 profile for real-time data. Every signed payload is a verifiable credential — not just a JSON object with a signature bolted on.

⬡

DataIntegrityProof

Every CDS event embeds a W3C DataIntegrityProof — the same proof format used in government credential issuance worldwide.

◎

ECDSA P-256

Signing algorithm: ecdsa-rdfc-2022. Elliptic curve P-256 — deterministic, compact, and hardware-accelerated on every modern platform.

◈

JSON-LD context

All identifiers are dereferenceable HTTP URIs. Data and meaning travel together. LLMs get typed, contextualised facts — not opaque strings.

◉

issuer DID anchor

Every credential names did:web:signed-data.org as issuer. The verification key is published at /keys/1 in the DID Document.

Security Model

Four signed properties

A DataIntegrityProof over the RDF dataset canonicalization (RDFC-1.0) covers exactly four claims — together they rule out every class of data-layer attack.

01

Source authenticity

The assertionMethod key in did:web:signed-data.org proves the issuer's identity — no impersonation.

02

Payload integrity

The proof covers the RDF-canonicalized dataset. Any modification — including whitespace — invalidates the signature.

03

Issuance timestamp

validFrom is part of the signed dataset. The credential cannot be back-dated or forward-dated without invalidating the proof.

04

Issuer accountability

The issuer DID resolves to a public key that can be audited, rotated, and revoked — creating a clear accountability chain.

Issuer Identity

did:web:signed-data.org

The issuer identity is a W3C DID — fully dereferenceable, human-readable, and anchored to a domain we control.

did:web:signed-data.org

Resolves to /.well-known/did.json. Contains the ECDSA P-256 verification key published at /keys/1. The assertionMethod field links the key to signing — verifiers have a single, canonical location for the public key.
